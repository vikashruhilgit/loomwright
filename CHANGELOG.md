# Changelog

Full release history for the AI Agent Manager plugin. The two most recent entries are also summarized at the top of `CLAUDE.md`; this file is the complete record.

> Entries are historical: the version and count claims in each describe **that** release, not the current state. The authoritative current version/counts live in `plugin.json`, `hooks.json`, and the `agents/`/`commands/`/`skills/` directories.

**v14.5.0 — Memory consume-path: `/dreaming` collects, distills, and writes-on-approval:** Closes the consume-half of project memory that v14.4.0 deferred to P4. `/dreaming` now (a) **collects** worker `WORKER_RESULT.memory_candidates[]` from worker summaries / session logs / done+failed briefs, (b) **distills bounded `LESSONS.md`** — at most **≤3 active entries per category** (a Reflexion-style sliding window, scored, oldest-evicted by the new repo-root sole writer **`write-lessons.sh`**), and (c) **writes accepted memory/LESSONS on per-item approval** via the repo-root sole writers (`write-project-memory.sh` / `write-lessons.sh`). This evolves `/dreaming` from strict propose-only to **propose-and-write-on-approval**, but it stays **human-gated** — no auto-write, and `CLAUDE.md` + legacy agent-memory remain paste-to-apply. **APPLY (reading LESSONS back at plan time) and auto-demotion are deferred to P5; LESSONS provenance is deferred to P5.** This added two scripts (`write-lessons.sh` + its test, not counted by the doc-currency gate) but **no new command / agent / skill / hook — still 19 hooks**, and **WORKER_RESULT stays `schema_version` 2**. Additive on top of v14.4.0.

**v14.4.0 — Project memory: Supervisor reads + Workers propose (P2b's deferred half):** Lands the participation that v14.3.0 deferred. The **Supervisor** now reads advisory project memory at the start of **Phase 1 ACQUIRE** (mirroring Launch Pad's Phase 3 step 0) via the provenance-gated `scripts/read-project-memory.sh` — advisory and strictly subordinate to `CLAUDE.md`, fail-safe (no memory / no hashing → proceeds silently). **Workers** now propose durable learnings via an **optional, additive `WORKER_RESULT.memory_candidates[]` field** — *propose-only, never write*: workers never touch `.supervisor/memory/`, the worktree-write ban (red-team F1) is intact, and promotion stays **human-gated**. This is purely additive: **no `schema_version` bump (WORKER_RESULT stays at 2)**, and **no new agent / command / skill / hook — still 19 hooks**. **Consume-path landed in v14.5.0:** `/dreaming` now collects worker `memory_candidates[]` (from summaries/logs/done+failed briefs), distills bounded `LESSONS.md`, and writes accepted items on per-item human approval — superseding this banner's "still-deferred P4" for collection + `LESSONS.md` reflection + `/dreaming` write (which stays human-gated, never auto). **Still deferred to P5:** LESSONS-APPLY (reading lessons back at plan time), auto-demotion, and LESSONS provenance. Additive on top of v14.3.0.

**v14.3.0 — Advisory project memory (P2b):** Adds agent-writable, cross-session **project memory** so the plugin stops re-discovering the codebase every run — built behind the red-team's guardrails. v1 layout under the gitignored `.supervisor/memory/`: `PROJECT_MEMORY.md` (advisory index, ≤200 lines) + `.provenance.jsonl` (hash-chained). (On-demand `topics/<topic>.md` detail files are a deferred P4 extension — v1 ships the index + provenance only.) **Sole writer** `scripts/write-project-memory.sh` **refuses any git-worktree CWD** (closes the F1 worktree-divergence FATAL — only repo-root callers write), hash-chains each entry, and enforces write-time eviction at the cap. **Read-side gate** `scripts/read-project-memory.sh` validates the chain and emits ONLY verified entries — an out-of-band poisoned line, or anything past a broken chain link, is dropped + logged to `.supervisor/logs/memory.log` (closes W1). Memory is **advisory and strictly subordinate to `CLAUDE.md`** (never an enforcement boundary), and **promotion is human-gated**: Launch Pad reads memory during Phase 3 ANALYZE and *proposes* new facts at Phase 6 for explicit user approval before any write. Hashing fails safe (no `sha256sum`/`shasum` → memory inert). v1 scope: **Launch Pad only** (Supervisor-read + Worker `memory_candidates[]` landed in v14.4.0; `LESSONS.md` reflection and `/dreaming` auto-write remain deferred P4). **No new agent / skill / command / hook — still 19 hooks.** Self-tested by `scripts/test-project-memory.sh` (worktree-guard, tamper-detection, poison-drop, eviction, gitignore). Design: `ai-agent-manager-plugin/docs/SPIKES/P2b_MEMORY_DESIGN.md`.

**v14.2.2 — Notification polish (patch):** Rounds out the v14.1.0 notification surface. `notify-desktop.sh` gains (1) **burst debounce** — coalesces rapid notification bursts (parallel hook fires / back-to-back gates) into one banner within a window set by `AI_AGENT_MANAGER_NOTIFY_DEBOUNCE` (default 5s; `0` disables; state in a gitignored `.supervisor/logs/.notify-debounce`), and (2) **Linux channel-detection** — only attempts `notify-send` when `DISPLAY`/`WAYLAND_DISPLAY` is set, skipping cleanly on headless boxes (the webhook is the headless channel). A terminal-bell tier was evaluated and deliberately omitted (a bell from a non-TTY hook goes nowhere useful). `/autonomous --notify` now **fails loud** at INIT step 0 when no webhook URL is resolvable (neither `$AI_AGENT_MANAGER_WEBHOOK_URL` nor `.supervisor/notify-config.json` → `.webhook_url`) — one warning instead of silently no-op'ing the webhook half (desktop banners unaffected). **No** hook-count (still 19), schema, agent, command, or skill change; `autonomous-loop` skill → 1.2.1. Additive patch on top of v14.2.1.

**v14.2.1 — Webhook + telemetry result-extraction fix (patch):** `send-webhook.sh` (the `supervisor_result` path) and `send-telemetry-core.sh` previously read the finishing subagent's output from a top-level **`result_block`** field that a real Claude Code **`SubagentStop`** payload never carries (verified by capturing a live payload). They now use the same robust resolution chain as `scripts/validate-launch-pad-result.py`: **`last_assistant_message`** (the real, observed inline field) → legacy `result_block` / `output` / `agent_output` (back-compat) → the last assistant message read from the transcript JSONL, preferring the subagent-scoped **`agent_transcript_path`** then the shared **`transcript_path`**. The pre-fix readers resolved `.result_block` to empty, so the supervisor-completion webhook (shipped v14.1.0) and every telemetry post were silently suppressed (status empty → POST skipped / `no_known_result_block`). The telemetry core also raw-scans the *resolved* result text for secrets before redaction (transcript-sourced text is not a payload field, so it would otherwise bypass the fail-closed privacy gate). `scripts/validate-launch-pad-result.py` (already on the `last_assistant_message` chain since v14.2.0) had its transcript fallback aligned to prefer the subagent-scoped `agent_transcript_path` too, so all three SubagentStop readers stay in sync. New dry-run self-tests: `scripts/test-webhook.sh` (real-field / legacy / transcript-fallback / suppression cases) and two discriminating `*-lastmsg-*` telemetry fixtures wired into `scripts/test-telemetry.sh`. Authoritative payload shape + resolution chain documented in `ai-agent-manager-plugin/docs/TELEMETRY.md` §"Result-text extraction". **No** hook-count (still 19), schema, agent, command, or skill change. Additive patch on top of v14.2.0.

**v14.2.0 — Concurrency & resume hardening:** Closes the `/autonomous` brief-detection footgun and adds crash/compact recovery. A new **`LAUNCH_PAD_RESULT`** result schema (`schema_version: 1`; fields `status` / `saved_brief_path` / `summary`; validated by `scripts/validate-launch-pad-result.py` on the `launch-pad-runner` SubagentStop hook, and in `--raw` mode by the autonomous-loop on the inline path) makes `saved_brief_path` the **primary** signal the loop uses to locate the saved brief — the fragile single-session `ls`-diff of `.supervisor/jobs/pending/` is demoted to a pre-v14.2.0 fallback, closing the concurrent-`/launch-pad` ambiguity. Launch Pad emits the block in a new **Phase 7** (every invocation, all outcome paths). Adds a **`SessionStart`** hook → `scripts/session-resume.sh`, which on `resume` / `clear` / `compact` injects a bounded (≤10k-char) `additionalContext` summary of in-progress/failed jobs + `state.md` tail + recent JSONL (silent on `startup`). **Two new hook entries** (`SessionStart`, `launch-pad-runner` SubagentStop) bring the count **17 → 19**. Additive on top of v14.1.0; no agent removed; no v14.0/v14.1 capability changed. (Reconciles the `LAUNCH_PAD_RESULT` + resume work from the closed PR #13.)

**v14.1.0 — Notification surface (desktop + phone, bidirectional-ready):** Adds OS-native pause notifications so a human knows the moment an autonomous run needs them. **Two new hook events** in `hooks.json` comprising **three new hook entries** (count **14 → 17**, counting by entry as the existing table does): `PreToolUse[AskUserQuestion]` → `scripts/notify-desktop.sh` (macOS `osascript` / Linux `notify-send` banner) **and** `scripts/send-webhook.sh` (paused-event POST) = 2 entries; plus `Notification` (matched to `permission_prompt`/`idle_prompt`/`elicitation_*`, deliberately excluding the noisy `auth_success`) → desktop banner = 1 entry. Fires on every plugin pause: Supervisor adjudication, `/autonomous` rubric gate, Plan Reviewer NEEDS_HUMAN, Launch Pad Phase 6, merge-and-continue. `send-webhook.sh` gains a **file-config fallback** (`.supervisor/notify-config.json` → `.webhook_url`) so config survives even when a URL exported only in `~/.zshrc` never reaches the non-interactive hook subprocess, plus an **ntfy-aware payload** (plain-text body + `Title`/`Priority`/`Tags` headers) for readable ntfy.sh phone pushes; the `--event-type gate` (v14.0.0) and `supervisor_result` paths are unchanged. Env vars: `AI_AGENT_MANAGER_DESKTOP_NOTIFICATIONS` (default on; `0` disables), `AI_AGENT_MANAGER_NOTIFY_SCOPE` (`plugin` default / `all`), `AI_AGENT_MANAGER_WEBHOOK_URL` (or the `.supervisor/notify-config.json` config file), `AI_AGENT_MANAGER_WEBHOOK_FORMAT=ntfy` (force plain-text payload for self-hosted ntfy). The desktop half needs no env var. The inbound reply leg (answer remotely → same session resumes) is Claude Code **Remote Control**. Additive on top of v14.0.0; no agent/command/skill changes. (Reconciles the work originally stranded on the closed PR #13 / `release/v13.1.0`.)

**v14.0.0 — Continuous autonomous mode with stacked PRs:** The `/autonomous` orchestration shell graduates from v13's opt-in re-planning to a **default multi-iteration loop** (cap 10, default 3) with **stacked-branch semantics**: iteration N+1 branches from `iterations[N].branch` rather than the integration base, so successive iterations form a reviewable stack. `--single-iteration` (or `--max-iterations 1`) preserves v13 single-PR behavior; `--no-stacked-branches` restores v13's branch-from-main cadence while keeping multi-iter mode active. New flags: `--notify` enables opt-in gate-event webhooks (rubric gate, adjudication gate, NO-GO override, Plan Review FAIL × 3) gated on `AI_AGENT_MANAGER_WEBHOOK_URL`; `--non-interactive-fallback` enables a per-gate fail-closed policy for CI / stdin-not-tty environments (rubric gate aborts with `rubric_gate_closed_non_interactive`; no-rubric `completed` runs return `done` with `no_rubric_in_non_interactive`; adjudication gate inherits Supervisor's `--non-interactive` policy if forwarded). Supervisor gains `--base-branch <ref>` and `--non-interactive` flags plus Phase 0/4/4.5 base-mismatch detection + cleanup, and emits new optional additive `branch_base` + `pr_state` fields on `SUPERVISOR_RESULT` (schema_version remains 1, both fields are optional). Context-Keeper gains atomic `set_flag` / `get_flag` / `clear_flag` operations writing under a new `## Phase Flags` section in `state.md` (consumed by autonomous-loop for stacked-branch handoff). `AUTONOMOUS_RUN` bumps to **schema_version 2** with nine new closed `status_reason` values covering the stacked-branch / non-interactive-fallback / gh-retry / no-rubric / deprecated-flag-conflict paths — v1 artifacts still accepted for the transition window. The authoritative list (with status-pairing) lives in `ai-agent-manager-plugin/docs/RESULT_SCHEMAS.md` §"AUTONOMOUS_RUN"; key examples include `non_interactive_without_fallback`, `iter_pr_base_mismatch`, `rubric_gate_closed_non_interactive`, `no_rubric_in_non_interactive`, `user_aborted_gh_retry`, `supervisor_base_branch_mismatch`. **W-NEW-3 spike PASSED pre-merge:** Code Reviewer and Rubric Grader both honor `DIFF-SCOPE OVERRIDE` inline directives on stacked-branch fixtures, removing the architectural blocker. Gate-event webhook payloads use **jq-only injection-safe construction** (no shell interpolation into JSON). v14 is **additive on top of v13**: no new agent, no hook count change (still 14), no removal of any v12.x / v13 capability. All v13.0.0 / v13.0.1 capabilities preserved (foreground-assisted gates bubble `AskUserQuestion` in-session; rubric-gate user-merge verification; `failed + inter_subtask_gap` Option C re-plan; webhook empty-payload suppression). All v12.2.0 capabilities preserved (Agent Teams graduation matrix, Outcomes Rubric, `/dreaming`, opt-in SubagentStop webhook hook), v12.1.0 documentation increments preserved (Memory Tool, Structured Outputs, SDK-only Advisor note), v12.0.0 reliability primitives intact (`provides`/`requires` contracts, pre-spawn verification gate, 4-option adjudication, effort-tier discipline, WORKER_RESULT schema_version 2 with `outputs_verified[]` + `outputs_gap`).
