# Changelog

Full release history for the AI Agent Manager plugin. The two most recent entries are also summarized at the top of `CLAUDE.md`; this file is the complete record.

> Entries are historical: the version and count claims in each describe **that** release, not the current state. The authoritative current version/counts live in `plugin.json`, `hooks.json`, and the `agents/`/`commands/`/`skills/` directories.

**v14.31.0 — Generic Launch Pad + Plan Reviewer prevention guardrails:** Three additive, repo-agnostic, **trigger-gated** prevention guardrails that catch a class of planning gaps before they surface as review churn — with no behavior change on ordinary briefs. **(1) Validator-owned-surfaces (Launch Pad Phase 3 ANALYZE):** when a task's acceptance depends on a generated/mechanical validation script (doc-currency / lint / codegen / consistency checker — stated generically, **never a hard-coded filename**), Launch Pad folds the surfaces that validator scans into the File Impact Map (or explicitly notes they may need updates), so a brief can no longer omit files a mechanical gate will later flag. **(2) Dirty-source-doc warning (Launch Pad Phase 1 VALIDATE):** when the specific source requirement/doc a brief is planned from is uncommitted in the working tree, Launch Pad emits a WARNING that the brief's provenance/staleness is unverified — scoped to that one source file, never a whole-tree dirty check. **(3) Plan Reviewer Criterion 15 (Canonical-List / Source-of-Truth Verification):** a conditional criterion that fires only when a brief asserts a canonical-source claim ("single source of truth" / "canonical list" / "exact names" / "authoritative") **and** cites a source — it then verifies the brief's restated values against that cited source and flags mismatches under a new `canonical_source` PLAN_REVIEW_RESULT issue category; absent that trigger it skips silently (the Criteria 11/13/14 conditional precedent). `canonical_source` is an **additive free-form** issue category (the `executable_acceptance` precedent) — **no `schema_version` bump** (PLAN_REVIEW_RESULT stays `schema_version: 1`). **No new agent / command / skill / hook (counts unchanged at 14 agents / 18 commands / 55 skills / 19 hooks).** Additive on top of v14.30.0.

**v14.30.0 — `/review-pr --until-mergeable` external-review-aware heal loop + churn-gated auto-postmortem:** Closes the long-standing gap that the entire review-and-heal machinery (Supervisor Phase 4.5, standalone `/review-pr` + `review-heal`, `/autonomous` EVALUATE) was a **closed internal loop** — it ran *our own* `code-reviewer` against the integrated diff and exited, never consulting external GitHub review state (`statusCheckRollup`, `reviews`, `reviewDecision`, review threads) and never re-acting on what the PR's reviewers/bots flagged. This adds an **opt-in, default-OFF** `--until-mergeable` mode to `/review-pr` + the `review-heal` skill that drains the *machine-speed* external signals and loops until the PR is mechanically merge-ready, **without ever merging and without ever waiting on a human**. **(1) External-state drain (AC1–AC3, AC14, AC15):** each round reads `gh pr view <url> --json statusCheckRollup,reviews,latestReviews,reviewDecision,mergeable,mergeStateStatus`, PLUS `gh api graphql` for review threads (`reviewThreads { isResolved, comments(first:1){ nodes{ author{ login __typename } } } }` — the thread state + author type that `gh pr view --json` does not expose), PLUS branch-protection **required-check discovery** (`gh api .../branches/{base}/protection/required_status_checks` or GraphQL `requiredStatusCheckContexts`). It auto-fixes the actionable BLOCKING/HIGH items via a `Task(general-purpose)` worker (allowlist Read/Write/Edit/Bash/Glob/Grep, **no Task**), `git push`es (**never `--force`**), and re-polls. **(2) Readiness semantics:** READY ⇔ **all required checks green AND no unresolved bot-authored review threads remain**. Required checks are gated from branch-protection metadata only (optional/non-required failing checks never block); when that metadata is unreadable the loop **fails CLOSED to `ESCALATED`** unless `--required-checks all-non-neutral` opts in. Threads count as READY-blockers only when the first comment's author is a bot (`author.__typename == "Bot"` or login `*[bot]`); **human approval, `REVIEW_REQUIRED`, and human-authored threads are surfaced but NEVER awaited**. On READY the loop stops and fires a best-effort desktop/webhook "ready to merge" notification. **(3) Bounds & anti-churn (AC4–AC6):** bounded by `--max-rounds N` (default 5) → `ESCALATED` + `gh pr comment` of remaining findings; an anti-churn fingerprint guardrail (`{file, issue_category, rule}`, trip on repeat-after-fix or `churn_rounds ≥ 2`) runs **one** deep "fix-the-class" self-review per trip; **no `gh pr merge` is ever invoked** in any terminal state (never-auto-merge invariant preserved). **(4) Churn-gated auto-postmortem (AC9–AC13):** on loop exit a new fail-safe dispatcher `scripts/dispatch-pr-postmortem.sh` (uncounted; mirrors `dispatch-pr-review.sh`) conditionally fires the existing read-only `/pr-postmortem <pr-url>` as a fresh detached `claude` process — **churn-gated** (`postmortem_churn_threshold`, default 2, via `--postmortem-churn-threshold` or `.supervisor/notify-config.json`; no-op on clean low-churn PRs), opt-out via `--no-auto-postmortem` / `auto_postmortem: false`. The `REVIEW_HEAL_RESULT.decision` is computed and emitted **before** dispatch and the dispatcher always exits 0 / only appends to `.supervisor/postmortem/results.jsonl`, so postmortem can **NEVER** alter the merge-ready result (the cardinal fail-safe guarantee). **(5) Schema + consumers (AC8, AC16):** `REVIEW_HEAL_RESULT` goes to **`schema_version: 2`** (adds `READY` decision + new fields + the GraphQL/required-check note in `docs/RESULT_SCHEMAS.md`; v1 still accepted), and the `/autonomous` consumer docs (`commands/autonomous.md`, `skills/autonomous-loop/SKILL.md`) list `READY` as additive, noting EVALUATE never emits it (it does not pass `--until-mergeable`) and treats unrecognized/`READY` as a terminal non-re-iterate state (degrades safely). **(6) Strictly additive (AC7):** `/review-pr` without `--until-mergeable` is byte-for-byte the existing diff-only review→fix→re-review loop. **No new agent / command / skill / hook (counts unchanged at 14 agents / 18 commands / 55 skills / 19 hooks — `dispatch-pr-postmortem.sh` is an uncounted script and `/pr-postmortem` already exists); no `SUPERVISOR_RESULT` schema_version bump.** Additive on top of v14.29.0.

**v14.26.0 — Close the requirement→brief→done loop (Beads-optional):** Closes a gap in the Beads-absent flow (Product Owner → `.supervisor/requirements/*.md` → Launch Pad → Supervisor): until now a Product Owner requirement file was write-once with no close-out marker, so done and not-done requirements looked identical on disk and operators had to hand-correlate slugs between `requirements/` and `jobs/done/`. This wires a provenance link end-to-end so a requirement that produced a successfully-completed brief is marked done automatically, with Beads remaining the sole source of truth when it is active. **Ownership split:** Launch Pad (the only agent that knows requirement→brief) owns the *provenance link*; the Supervisor completion tail (the only thing alive at done-time, already the sole writer of job lifecycle) owns the *close-out*. **(1) Producer** — `agents/launch-pad.md` Phase 2 step 0 records the resolved `.supervisor/requirements/*.md` path as `source_requirement` (only for requirement-file inputs; unset for literal-string goals or repo files outside the handoff dir), and Phase 5 PACKAGE emits a `- **Source requirement:** {repo-root-relative path}` line under the brief's `## Environment` only when resolved (omitted entirely otherwise — no placeholder); the optional field is documented in the `skills/supervisor-readiness/SKILL.md` brief template. **(2) Consumer** — `agents/supervisor.md` Phase 4.5 SELF_HEAL completion-tail gains step 2.5: on a successful outcome (PASS / loop-skipped / ESCALATED) **and only when Beads is inactive** (`test -d .beads && bd --version`), it reads the provenance pointer, path-safety-checks it (must resolve **under `.supervisor/requirements/`** AND pass `test -f` — guards traversal/injection via the brief field), and stamps an **idempotent** `## Status` block (`Status`, `Completed`, `Brief`, `PR`) on the originating requirement file **in place** (stamp-not-move handles the multi-brief case; replace-not-duplicate so the latest close-out wins). The `Status` value **mirrors the brief `## Outcome` granularity** — PASS / loop-skipped → `done`; ESCALATED → `done_with_escalation` plus a `- **Heal:**` reason + remaining-issue line — so an escalated requirement is not indistinguishable from a clean pass. The whole step is a **fail-safe runtime side-effect emitter**: any error (unreadable brief, missing file, write failure, malformed pointer, path outside the handoff dir) is a logged no-op that NEVER propagates to `SUPERVISOR_RESULT.status` and never fails the run (per the CLAUDE.md bimodal "side-effect emitters fail SAFE" invariant); a `failed` / abort / checkpoint / base-mismatch / invariant-violation run never reaches it, so a failed run never marks a requirement done. **(3) Docs** — `docs/RESULT_SCHEMAS.md` documents both the brief `Source requirement` field and the requirement `## Status` close-out convention (advisory / Beads-absent-only, mirrors the brief `## Outcome` pattern). **Prompt/doc-only; no new agent / command / skill / hook; no schema_version bumps. Counts unchanged at 14 agents / 18 commands / 54 skills / 19 hooks.** Additive on top of v14.25.1.
